## 第一章：项目概览

### 1.1 这个项目做什么？

arinelbot 是一个 **B站动态/直播推送 QQ 机器人**。它的工作流程是：

1. 用户在 QQ 群里发送 `/sub <UP主ID>` 订阅某个 UP 主
2. 机器人每隔 8 秒检查该 UP 主是否有新动态
3. 如果有新动态，渲染成图片卡片，推送到 QQ 群
4. 同样，每隔 10 秒检查直播状态，开播时推送通知

### 1.2 为什么用 Clean Architecture？

想象一下你写了一个"大泥球"程序——所有代码混在一起：

```
# 反面教材：所有东西混在一起
async def check_dynamic():
    conn = sqlite3.connect("data.db")        # 数据库代码
    cursor = conn.execute("SELECT ...")
    response = httpx.get("https://bilibili.com/...")  # API 调用
    img = skia.render(response.json())        # 渲染图片
    bot.send_group_msg(group_id, img)         # 发送消息
```

这段代码的问题：

| 问题 | 说明 |
|------|------|
| **无法测试** | 想测试？必须连真实数据库、真实 B站 API、真实 QQ 机器人 |
| **无法替换** | 想换数据库？所有代码都要改 |
| **职责不清** | 一段代码同时做 5 件事，出了 bug 不知道在哪 |

Clean Architecture 的解决方案是 **分层**，每层只做一件事：

```
┌─────────────────────────────────────────────┐
│  Presentation (表现层)                       │  ← 用户看到的：命令、定时任务
├─────────────────────────────────────────────┤
│  Application (应用层)                        │  ← 业务流程：订阅、推送、查询
├─────────────────────────────────────────────┤
│  Infrastructure (基础设施层)                  │  ← 技术实现：数据库、API、渲染
├─────────────────────────────────────────────┤
│  Domain (领域层)                             │  ← 核心概念：实体、事件、接口
└─────────────────────────────────────────────┘
```

**核心规则：依赖只能从外向内，内层永远不知道外层的存在。**

Domain 层定义"我需要一个能存储数据的东西"（接口），Infrastructure 层实现"我用 SQLite 来存"（实现）。如果将来要换成 MySQL，只需要换 Infrastructure 层，Domain 和 Application 完全不用动。

## 第二章：Python 基础概念速查

### 2.1 async/await 异步编程

**同步 vs 异步：**

```python
# 同步：一个一个来，等前一个做完才做下一个
result1 = fetch_from_api()     # 等 2 秒
result2 = query_database()     # 等 1 秒
# 总耗时：3 秒

# 异步：等的时候可以做别的事
result1 = await fetch_from_api()   # 发出请求，等的时候去干别的
result2 = await query_database()   # 发出查询，等的时候去干别的
# 总耗时：约 2 秒（两个操作有重叠）
```

**在项目中的例子：**

```python
# arielbot/application/push/check_jobs.py:28
async def run(self) -> None:
    follow_dynamic_list = await self._api.get_follow_dynamics()
    #                          ↑
    #    await 表示"等这个异步操作完成，但等待期间不阻塞其他任务"
```

**关键点：**
- `async def` 定义一个异步函数（协程）
- `await` 调用一个异步函数
- 只有在 `async def` 函数内部才能用 `await`
- Python 的异步是 **协作式** 的——一个任务必须主动 `await` 才能让出控制权

### 2.2 @dataclass 装饰器

```python
# 没有 @dataclass，你需要手写很多样板代码：
class SubTarget:
    def __init__(self, uid: str, nickname: str):
        self.uid = uid
        self.nickname = nickname
    def __repr__(self):
        return f"SubTarget(uid={self.uid}, nickname={self.nickname})"
    def __eq__(self, other):
        return self.uid == other.uid and self.nickname == other.nickname

# 用 @dataclass，自动生成上面所有代码：
from dataclasses import dataclass

@dataclass
class SubTarget:
    uid: str
    nickname: str
# 就这么简单！__init__、__repr__、__eq__ 全部自动生成
```

**在项目中：**


### 2.3 ABC 抽象基类

ABC（Abstract Base Class）是 Python 的"接口"机制。它定义一个"契约"——所有实现这个接口的类 **必须** 实现指定的方法。

```python
from abc import ABC, abstractmethod

# 定义接口（契约）
class Animal(ABC):
    @abstractmethod
    def speak(self) -> str:
        ...    # ... 是占位符，表示"这里不写实现"

# 实现接口
class Dog(Animal):
    def speak(self) -> str:    # 必须实现 speak，否则报错
        return "汪汪！"

# 不能实例化抽象类
# animal = Animal()  # 报错！TypeError
dog = Dog()           # OK
dog.speak()           # "汪汪！"
```

**在项目中：**

```python
# arielbot/domain/interfaces/repository.py
class SubTargetRepository(ABC):
    @abstractmethod
    async def get(self, uid: str) -> Optional[SubTarget]:
        ...    # 只定义"需要一个 get 方法"，不写怎么实现

    @abstractmethod
    async def save(self, uid: str, nickname: str) -> None:
        ...    # 只定义"需要一个 save 方法"，不写怎么实现
```

Infrastructure 层提供具体实现：

```python
# arielbot/infrastructure/repositories/target_repository.py
class SqlSubTargetRepository(SubTargetRepository):   # 实现接口
    async def get(self, uid: str) -> Optional[SubTarget]:
        async with self._db.transaction() as cursor:
            await cursor.execute("SELECT uid, nickname FROM subTarget WHERE uid=?", (uid,))
            row = await cursor.fetchone()
            if row is None:
                return None
            return SubTarget(uid=row[0], nickname=row[1])
```

**为什么用 ABC？**
- Application 层只依赖 `SubTargetRepository` 这个接口
- 它不知道也不关心底层是 SQLite、MySQL 还是内存存储
- 测试时可以用假的实现（Mock），不需要真实数据库

### 2.5 asyncio.Queue 异步队列

```python
import asyncio

queue = asyncio.Queue(maxsize=100)   # 最多放 100 个元素

# 生产者：往队列里放
await queue.put("事件A")        # 队列满时会等待
queue.put_nowait("事件B")       # 队列满时直接报错（不等待）

# 消费者：从队列里取
event = await queue.get()       # 队列空时会等待
```

**为什么用队列？**
- 生产者（CheckJob）和消费者（Handler）解耦
- 生产者只管"发布事件"，不关心谁处理、怎么处理
- 队列起到缓冲作用，防止高负载时丢失事件

### 2.6 @asynccontextmanager 上下文管理器

```python
from contextlib import asynccontextmanager

# 定义上下文管理器
@asynccontextmanager
async def open_file(path):
    f = await aio_open(path)
    try:
        yield f          # yield 把值给到 as 后面的变量
    finally:
        await f.close()  # 无论如何都关闭

# 使用：自动管理资源的打开和关闭
async with open_file("test.txt") as f:
    content = await f.read()
# 离开 async with 块后，f 自动关闭
```

**在项目中：**

```python
# arielbot/infrastructure/database.py:26
@asynccontextmanager
async def transaction(self) -> AsyncIterator[aiosqlite.Cursor]:
    conn = await aiosqlite.connect(self._db_path)   # 打开连接
    cursor = await conn.cursor()
    # ... 设置 PRAGMA ...
    await cursor.execute("BEGIN")                     # 开始事务
    try:
        yield cursor                                  # 把 cursor 给使用者
        await conn.commit()                           # 成功则提交
    except Exception:
        await conn.rollback()                         # 失败则回滚
        raise                                         # 重新抛出异常
    finally:
        await cursor.close()                          # 清理资源
        await conn.close()                            # 关闭连接

# 使用：一行代码搞定事务管理
async with self._db.transaction() as cursor:
    await cursor.execute("SELECT * FROM subTarget")
    rows = await cursor.fetchall()
# 离开 async with 后，自动 commit/rollback + 关闭连接
```

## 第三章：Domain 层 — 领域核心

Domain 层是整个项目的 **心脏**。它定义了：

- **实体（Entities）**：核心数据结构
- **事件（Events）**：系统中发生的"事情"
- **接口（Interfaces）**：对外部能力的抽象描述

Domain 层 **零依赖**——不导入任何框架（NoneBot、aiosqlite、httpx 等）。这是 Clean Architecture 最重要的原则。

### 3.1 entities.py — 数据实体

```python
# arielbot/domain/entities.py
from dataclasses import dataclass

@dataclass
class SubTarget:
    uid: str          # B站用户 ID（如 "12345"）
    nickname: str     # 用户昵称（如 "某UP主"）
```

**SubTarget** 代表一个被订阅的 B站 UP 主。它只有两个字段——ID 和昵称。

```python
@dataclass
class SubChannel:
    uid: str           # 订阅的 UP 主 ID
    group_id: int      # QQ 群号
    bot_id: int        # 机器人账号 ID
    live_active: bool  # 这个群是否开启了该 UP 主的直播推送
    dyn_active: bool   # 这个群是否开启了该 UP 主的动态推送
```

**SubChannel** 代表一条"订阅关系"——某个 QQ 群通过某个 Bot 订阅了某个 UP 主。这是多对多关系的中间表。

```python
@dataclass
class BotStatus:
    bot_id: int        # 机器人账号 ID
    group_id: int      # QQ 群号
    push_active: bool  # 推送总开关（管理员可以关闭）
    bot_active: bool   # Bot 是否在线（自动管理）
```

**BotStatus** 记录机器人在某个群的状态。

### 3.2 events.py — 领域事件

事件（Event）描述"系统中发生了什么事"。发布者不关心谁处理，处理者不关心谁发布的。

```python
# arielbot/domain/events.py

@dataclass
class DynamicDetected:
    """当检测到新动态时，发布这个事件"""
    dynamic: object                # 原始动态对象（来自 B站 API）
    dyn_id: str                    # 动态 ID
    uname: str                     # UP 主昵称
    targets: List[Tuple[int, int]] # 推送目标列表 [(群号, Bot ID), ...]
    rendered_image: bytes          # 渲染好的图片

@dataclass
class LiveStatusChanged:
    """当直播状态变化时，发布这个事件"""
    uid: str
    uname: str
    room_id: str                   # 直播间 ID
    title: str                     # 直播标题
    cover_url: str                 # 封面图 URL
    is_live: bool                  # 是否正在直播
    targets: List[Tuple[int, int]]

@dataclass
class BotConnected:
    """当 Bot 连接时发布"""
    bot_id: int

@dataclass
class BotDisconnected:
    """当 Bot 断开时发布"""
    bot_id: int

@dataclass
class BotShutdown:
    """当系统关闭时发布"""
    pass                           # 无数据，纯信号
```

**事件驱动的好处：**
- CheckJob 只管"检测到新动态了！"（发布事件）
- Handler 只管"有新动态？我来发消息！"（处理事件）
- 两者完全不知道对方的存在，通过 EventBus 连接

### 3.3 interfaces/repository.py — 仓储接口

Repository 模式把"数据存储"抽象成一组方法调用。Application 层不需要知道数据存在哪里。

```python
# arielbot/domain/interfaces/repository.py

class SubTargetRepository(ABC):
    @abstractmethod
    async def get(self, uid: str) -> Optional[SubTarget]:
        """根据 UID 获取 UP 主信息，找不到返回 None"""
        ...

    @abstractmethod
    async def save(self, uid: str, nickname: str) -> None:
        """保存 UP 主信息"""
        ...

    @abstractmethod
    async def update(self, nickname: str, uid: str) -> None:
        """更新 UP 主昵称"""
        ...
```

项目定义了 5 个 Repository 接口：

| 接口 | 管理什么 | 关键方法 |
|------|---------|---------|
| `SubTargetRepository` | UP 主信息 | get, save, update |
| `SubChannelRepository` | 订阅关系 | get, save, update, delete, find_push_targets |
| `BotStatusRepository` | Bot 状态 | get, save, update_push, deactivate_all |
| `CookieRepository` | 登录凭证 | get, save, update, clear |
| `DynCacheRepository` | 动态缓存 | exists, save, find, archive_old_dynamics |

## 第四章：Application 层 — 业务逻辑

### 4.1 auth_service.py — 扫码登录

### 4.3 query_service.py — Cache-Aside 缓存模式

```python
async def _get_or_fetch_dynamic(self, dyn_id: str) -> Optional[object]:
    # 1. 先查缓存
    cached = await self._dyn_cache_repo.find(dyn_id)
    if cached:
        return pickle.loads(cached)           # 缓存命中，直接返回

    # 2. 缓存未命中，从 API 获取
    dynamic = await self._api.get_dynamic_by_id(dyn_id)
    if dynamic is None:
        return None

    # 3. 存入缓存
    await self._dyn_cache_repo.save(
        dyn_id, dynamic.header.name, pickle.dumps(dynamic)
    )
    return dynamic
```

这是经典的 **Cache-Aside** 模式：先查缓存 → 没有就查数据库/API → 存入缓存。

**run_in_executor 异步化同步 I/O：**

```python
async def get_help_image(self) -> bytes:
    loop = asyncio.get_running_loop()
    if os.path.exists(_HELP_CACHE_PATH):
        # 文件读取是同步操作，放到线程池执行
        return await loop.run_in_executor(None, _read_help_cache)
    # ...
    img_bytes = await self._help_renderer.render()
    # 文件写入也放到线程池
    await loop.run_in_executor(None, _write_help_cache, img_bytes)
    return img_bytes

# 线程池函数必须定义在模块级别
def _read_help_cache() -> bytes:
    with open(_HELP_CACHE_PATH, "rb") as f:
        return f.read()
```

**为什么需要 run_in_executor？**
- `open().read()` 是同步 I/O，会阻塞事件循环
- 如果不放到线程池，整个程序在读文件时"卡住"，无法处理其他请求
- `run_in_executor(None, fn)` 把 `fn` 放到默认线程池执行，不阻塞主事件循环

```python
class DynCheckJob:
    async def run(self) -> None:
        # 1. 获取关注列表的最新动态
        follow_dynamic_list = await self._api.get_follow_dynamics()
        if follow_dynamic_list is None:
            return

        for dynamic in follow_dynamic_list:
            # 2. 检查缓存：这个动态是不是已经处理过了？
            cached = await self._dyn_cache_repo.exists(dynamic.message_id)
            if cached:
                continue    # 跳过已处理的

                dynamic.header.mid
            )
            if not targets:
                # 没人订阅这个 UP 主，只缓存不推送
                await self._dyn_cache_repo.save(...)
                continue

            # 4. 渲染图片 + 发布事件
            rendered = await self._dyn_renderer.render(dynamic)
            await self._dyn_cache_repo.save(...)
            await self._event_bus.publish(DynamicDetected(
                dynamic=dynamic,
                dyn_id=dynamic.message_id,
                uname=dynamic.header.name,
                targets=targets,
                rendered_image=rendered,
            ))
```

**LiveCheckJob（每 10 秒执行一次）——边沿检测：**

```python
class LiveCheckJob:
    def __init__(self, ...):
        self._live_cache: dict[str, int] = {}   # 内存缓存：上次的直播状态

    async def run(self) -> None:
        uid_list = await self._channel_repo.find_all_subscribed_uids()
        check_result = await self._api.get_room_info_by_uids(uid_list)

        for k, v in check_result.items():
            current = 1 if v["live_status"] == 1 else 0   # 当前状态
            prev = self._live_cache.get(k, current)        # 上次状态（首次用当前值）
            self._live_cache[k] = current                  # 更新缓存

            if prev == current:
                continue        # 状态没变，跳过
            if prev == 1:
                continue        # 从开播→下播，不通知（只通知开播）

            # 状态从 0→1：开播了！发布事件
            await self._event_bus.publish(LiveStatusChanged(...))
```

**边沿检测**：不是"是否在直播"（状态），而是"是否刚刚开播"（变化）。`prev=0, current=1` 才触发，避免重复通知。

### 5.2 database.py — 数据库管理

```python
class DatabaseManager:
    def __init__(self, db_path: Optional[str] = None):
        self._db_path = db_path or path.join(getcwd(), "data.sqlite")
        self._initialized = False     # 懒初始化标记

    @asynccontextmanager
    async def transaction(self) -> AsyncIterator[aiosqlite.Cursor]:
        # 1. 打开连接
        conn = await aiosqlite.connect(self._db_path)
        cursor = await conn.cursor()

        # 2. 设置 PRAGMA（性能优化）
        await cursor.execute("PRAGMA journal_mode = WAL")     # 写前日志模式
        await cursor.execute("PRAGMA busy_timeout = 5000")    # 锁等待 5 秒
        await cursor.execute("PRAGMA foreign_keys = ON;")     # 启用外键

        # 3. 首次使用时创建表
        await self._ensure_initialized(conn)

        # 4. 开始事务
        await cursor.execute("BEGIN")
        try:
            yield cursor              # 使用者在这里操作数据库
            await conn.commit()       # 成功 → 提交
        except Exception:
            await conn.rollback()     # 失败 → 回滚
            raise                     # 重新抛出异常
        finally:
            await cursor.close()      # 清理
            await conn.close()
```

**PRAGMA 设置解释：**

| PRAGMA | 作用 |
|--------|------|
| `journal_mode = WAL` | 写前日志模式，允许读写并发，提升性能 |
| `busy_timeout = 5000` | 数据库被锁时等待 5 秒再报错 |
| `foreign_keys = ON` | 启用外键约束（SQLite 默认关闭） |

**数据库迁移：**

```python
async def _migrate(self, cursor: aiosqlite.Cursor) -> None:
```

### 5.3 event_bus.py — 事件总线实现

```python
class SimpleEventBus(EventBus):
    def __init__(self):
        self._handlers: Dict[Type, List[Callable]] = {}  # 事件类型 → 处理器列表
        self._queue: asyncio.Queue = asyncio.Queue(maxsize=100)
        self._task: Optional[asyncio.Task] = None

    def subscribe(self, event_type: Type, handler: Callable) -> None:
        # 注册：DynamicDetected → [DynPushHandler.handle]
        if event_type not in self._handlers:
            self._handlers[event_type] = []
        self._handlers[event_type].append(handler)

    async def publish(self, event: Any) -> None:
        # 发布：把事件放入队列（非阻塞）
        try:
            self._queue.put_nowait(event)
        except asyncio.QueueFull:
            logger.warning(f"EventBus queue full, dropping {type(event).__name__}")

    async def _dispatch_loop(self) -> None:
        # 后台循环：从队列取事件，分发给处理器
        while True:
            event = await self._queue.get()         # 阻塞等待事件
            for event_type, handlers in self._handlers.items():
                if isinstance(event, event_type):   # 类型匹配
                    for handler in handlers:
                        try:
                            await handler(event)    # 调用处理器
                        except Exception as e:
                            logger.error(f"Handler error: {e}")
                            # 一个处理器出错不影响其他处理器
```

**isinstance 路由**：用 `isinstance` 检查事件类型，把事件路由到对应的处理器。这是 Python 的运行时多态。

```python
            FROM subTarget t1                          -- UP 主表

            INNER JOIN botStatus t3 ON t2.groupId = t3.groupId AND t2.bot = t3.bot
```

```python
class CommandRegistry:
    _matchers: Dict[str, Matcher] = {}    # 类变量：所有注册的命令

    @classmethod
    def register(cls, name: str, *,
                 permission: Optional[Permission] = None,
                 aliases: Optional[Set[str]] = None,
                 rule: Optional[Rule] = None) -> Matcher:
        matcher = on_command(name, permission=permission,
                             aliases=aliases, rule=rule)
        cls._matchers[name] = matcher
        return matcher

    @classmethod
    def register_all(cls, container: Any) -> None:
        bot_is_active = make_bot_is_active_rule(container.bot_status_service)

        # 注册 /login 命令
        login_matcher = cls.register("login", permission=SUPERUSER, aliases={"登录"})
        @login_matcher.handle()
        async def _(bot: Bot, event: GroupMessageEvent):
            handler = make_login_handler(container.auth_service)
            await handler(bot, event)
            await login_matcher.finish()

        # 注册 /sub 命令
        sub_matcher = cls.register("sub",
            permission=SUPERUSER | GROUP_ADMIN | GROUP_OWNER,
            rule=bot_is_active, aliases={"订阅"},
        )
        @sub_matcher.handle()
        async def _(event: GroupMessageEvent, args: Message = CommandArg()):
            handler = make_sub_handler(container.sub_service, sub_matcher)
            await handler(event, args)

        # ... 其他命令 ...
```

```python
    @driver.on_bot_connect
    async def _(bot: Bot) -> None:
        await event_bus.publish(BotConnected(bot.self_id))

    @driver.on_bot_disconnect
    async def _(bot: Bot) -> None:
        await event_bus.publish(BotDisconnected(bot.self_id))
```

**闭包（Closure）**：`make_bot_is_active_rule` 返回的 `bot_is_active` 函数"记住"了外部的 `bot_status_service` 变量。这就是闭包——内部函数可以访问外部函数的变量。

### 6.4 scheduler/__init__.py — 定时任务

```python
def register_scheduled_jobs(scheduler: AsyncIOScheduler,
                            dyn_check_job: DynCheckJob,
                            live_check_job: LiveCheckJob,

    @scheduler.scheduled_job("cron", hour="3", id="dyn_archive", max_instances=1)
```

## 第八章：设计模式速查表

| 模式 | 一句话解释 | 项目中的位置 |
|------|-----------|-------------|
| **Repository** | 把数据存储抽象成方法调用 | `domain/interfaces/repository.py` → `infrastructure/repositories/` |
| **Event Bus** | 发布者不关心谁处理，处理者不关心谁发布 | `domain/interfaces/event_bus.py` → `infrastructure/event_bus.py` |
| **Dependency Injection** | 通过构造函数传入依赖，而不是自己创建 | `Container.__init__()` 组装所有依赖 |
| **Strategy** | 定义接口，运行时选择具体实现 | `DynRenderer` / `SubListRenderer` / `HelpRenderer` |
| **Adapter** | 把外部接口转换成内部接口 | `bili_api.py`、`bot_client.py`、`renderer.py` |
| **Factory Method** | 一个方法创建并返回对象 | `CommandRegistry.register()` 创建命令 |
| **Cache-Aside** | 先查缓存 → 没有就查源 → 存入缓存 | `QueryService._get_or_fetch_dynamic()` |
| **Unit of Work** | 一个事务内完成所有操作，要么全成功要么全回滚 | `DatabaseManager.transaction()` |
| **Observer** | 对象状态变化时通知所有观察者 | EventBus 的 subscribe/publish |
| **Composition Root** | 整个应用只有一个地方知道所有具体类 | `Container` + `plugins/Core/__init__.py` |
| **Port/Adapter** | 内层定义接口（Port），外层提供实现（Adapter） | Domain interfaces → Infrastructure adapters |
| **Closure** | 内部函数记住外部变量 | `make_bot_is_active_rule()` 返回的函数 |

## 第九章：常见疑问 Q&A

### Q1: 为什么要分四层？直接写不行吗？

小项目当然可以直接写。但当项目变大：
- 你想写测试时，发现必须连真实 B站 API
- 你想换数据库时，发现要改 100 个地方
- 你出了 bug，不知道是数据库问题还是业务逻辑问题

分层后：
- 测试时注入 Mock，不需要真实 API
- 换数据库只改 Infrastructure 层
- 每层职责清晰，定位问题快

### Q2: ABC 和普通类有什么区别？

```python
# 普通类：可以实例化
class Dog:
    def speak(self):
        return "汪"

dog = Dog()   # OK

# 抽象类：不能实例化，只能被继承
from abc import ABC, abstractmethod
class Animal(ABC):
    @abstractmethod
    def speak(self):
        ...

# animal = Animal()  # 报错！TypeError

class Dog(Animal):
    def speak(self):    # 必须实现，否则实例化时报错
        return "汪"

dog = Dog()   # OK
```

ABC 的作用是 **定义契约**——"如果你想当一个 Repository，你必须实现 get/save/update 方法"。

### Q3: async/await 和 threading 有什么区别？

| | async/await | threading |
|---|------------|-----------|
| **并发模型** | 协作式（一个任务主动让出） | 抢占式（操作系统切换） |
| **适用场景** | I/O 密集（网络请求、数据库查询） | CPU 密集（计算、渲染） |
| **开销** | 很轻量（一个协程约几 KB） | 较重（一个线程约 1MB） |
| **切换** | 只在 `await` 时切换 | 随时可能被切换 |
| **竞态条件** | 不会（单线程） | 会（需要锁） |

项目中：
- 网络请求、数据库操作 → `async/await`
- Skia 渲染（CPU 密集）→ `run_in_executor` 放到线程池

### Q4: 为什么用 pickle 而不是 JSON？

```python
# JSON 只能序列化基本类型
json.dumps({"name": "test", "id": 123})   # OK

# pickle 可以序列化几乎任何 Python 对象
pickle.dumps(dynamic_object)   # 包含复杂嵌套对象、自定义类等
```

B站动态对象（来自 `dynamicadaptor` 库）是复杂的嵌套结构，用 JSON 序列化太麻烦。pickle 虽然有安全风险，但在这个场景下（自己写入、自己读取）是可接受的。

### Q5: Container 为什么要手动组装？

项目用的是 **手动依赖注入**（Manual DI），没有用 DI 框架。好处是：
- 代码简单，没有魔法
- 容易理解依赖关系
- 不需要学习额外框架

缺点是 Container 会比较长。对于这个规模的项目，手动 DI 是最佳选择。

### Q6: 为什么 Repository 返回 Optional 而不是抛异常？

```python
# 方案 A：返回 Optional
async def get(self, uid: str) -> Optional[SubTarget]:
    # 找到返回 SubTarget，找不到返回 None

# 方案 B：抛异常
async def get(self, uid: str) -> SubTarget:
    if not found:
        raise NotFoundError(uid)
```

项目选择方案 A，因为"找不到"是正常情况（用户可能还没订阅），不是错误。用异常处理正常流程会让代码难以理解。

调用方可以简洁地处理：

```python
target = await self._target_repo.get(uid)
if target:
    # 找到了，继续
else:
    # 没找到，走创建流程
```

### Q7: 为什么每个 Repository 方法都开一个新连接？

```python
async def get(self, uid: str) -> Optional[SubTarget]:
    async with self._db.transaction() as cursor:   # 每次新连接
        ...
```

这是 `aiosqlite` 的特性——SQLite 是文件数据库，不支持真正的连接池。每次事务开一个连接是 SQLite 的标准做法。WAL 模式确保了读写并发不会互相阻塞。
